# Telegram-бот — агрегатор/фильтр вакансий (учебный проект)

> Контекст для будущих сессий Claude Code. Кратко. Полная постановка — у владельца.

## Что это
Учебный проект, **не продукт и не монетизация**. Единственный пользователь — владелец.
Цель — впервые руками пощупать **проектирование/архитектуру, бэкенд, БД, работу с AI API**.
hh.ru — осознанно учебный полигон. Боль: листание полотен неподходящих вакансий.

## Кто владелец
Senior QA, переходит в автоматизацию/SDET. Сильная зона — тесты (`pytest`).

## Режим работы: вайб-кодинг (СОБЛЮДАТЬ СТРОГО)
- **Код пишет Claude.** Владелец отвечает за верхнеуровневые решения (что делаем, какая
  архитектура, следующий шаг) и за разбор готового кода.
- Цель владельца — учиться **верхнеуровневым решениям** и **читать/понимать код**, а не
  набивать его руками.
- **Перед/после кода — короткое объяснение «что и зачем написал»**, чтобы владелец понял
  diff. Теория идёт как сопровождение кода, а не как задание написать самому.
- Прямо, по делу, без воды. Лучше 5–7 проработанных пунктов, чем 30 «на всякий».
- **Один путь решения, без вариантов A/B/C.**
- Ошибку в подходе/решении называть прямо.
- HTTP/REST/API-концепции объяснять НЕ надо — только специфику hh/инструментов.
- Объяснения — на русском. Контекст «зачем» важен.
- Работаем через VS Code (расширение `anthropic.claude-code`): правки видны как diff.

## Архитектура
Трёхмодульный конвейер: **источник → фильтр → доставка**. Связывает только `pipeline.py`;
модули НЕ импортируют друг друга. **Ядро — единая Pydantic-модель вакансии = контракт
между модулями.** Каждый источник приводит свой формат к этой модели.

```
src/
  models/     # Pydantic-модель вакансии (от неё зависят все)
  sources/    # hh (сейчас), telegram (потом)
  filters/    # фильтр-правила (общий для всех источников)
  delivery/   # консоль (сейчас), telegram-бот (потом)
  pipeline.py # источник → фильтр → доставка
config/       # критерии фильтрации как ДАННЫЕ, не в коде
tests/        # pytest
scripts/      # разовые скрипты (разведка Ит.0) — НЕ часть конвейера
.gitignore    # .venv/  __pycache__/  .env
```

## Итерации (порядок 0→1 жёсткий, 2–4 переставляемы)
0. **Разведка hh API** — одноразовый скрипт в `scripts/`, увидеть СЫРОЙ ответ.
1. **Ядро конвейера** на одном источнике: Pydantic-модель + фильтр-конфиг
   (ключевые/стоп-слова + порог зарплаты) + доставка В КОНСОЛЬ.
2. **Telegram-доставка** (`aiogram`, бот через @BotFather, сводка в личку).
3. **Интерактивные фильтры** в боте через FSM — слой поверх фильтр-функции.
4. **AI-слой ранжирования** (1 HTTP-запрос к LLM «подходит + причина»).

Не путать: ранжирование (ИИ, Ит.4) ≠ контроль объёма выдачи (пагинация + параметры hh).

## СТАТУС: итерации 0-5 закрыты, бот работает

Источник hh закрыт (403 анонимно, поддержка соискателей прекращена 15.12.2025) —
заменён на Telegram-каналы через Telethon. Смена источника не сломала остальное:
менялся только `sources/`. Ровно ради этого модули и разделяли.

### Что есть
- `sources/` Telethon (техаккаунт) -> `parsing/` регулярки -> `enrichment/` Haiku
  (structured output + prompt caching) -> `filters/` правила + ИИ-оценка ->
  `storage/` SQLite (дедуп + настройки) -> `delivery/` aiogram-бот.
  Связывает `pipeline/`.
- Критерии — из резюме (`profile/resume_parser.py`), а не из конфига. Фильтр
  принимает `Criteria` параметром и не знает, откуда они.
- Два процесса: `scripts/daily_fetch.py` (крон, раз в сутки) и
  `scripts/run_bot.py` (бот, слушает кнопки).

### Режим работы: pull с тихим дайджестом
Крон собирает раз в сутки и шлёт ОДНО тихое сообщение "доступно N новых".
Показ вакансий — только по кнопке и всегда из БД (ни Telethon, ни ИИ).
Кнопка НЕ запускает конвейер — иначе вернётся минута ожидания.

### Проектные ограничения (осознанные, не баги)
- Один пользователь, один профиль.
- Опрос каналов раз в сутки — риск бана Telethon-аккаунта.
- Бот не пушит вакансии по одной, только дайджест.
- Batch API не используется: латентность до 24ч, а дедуп сводит поток к
  ~80 постам/сутки, где экономия — центы.

### Замеры (на 300 постах, 76 дней)
- Поток канала ~118 постов/мес; на пороге 270к проходит ~8 вакансий/мес.
- Одного канала мало — нужно 2-3, задаются через `CHANNELS` в `.env`.
- Установившийся прогон ~3 с, экономия на кэше 88%, ~$0.30 за 300 постов.

### Долги
- Клиент Anthropic продублирован в `enrichment`, `filters`, `profile`.
- `seen_posts` растёт вечно; смена критериев не переоценивает старые посты.
- `pending_deliveries`/`delivered_at` — рудимент push-модели Итерации 4.
- Стек из резюме шире ручного (12 технологий против 5): резюме говорит
  "что умею", а фильтру нужно "с чем хочу работать".
- Деплой на VPS не делался — всё локально.
