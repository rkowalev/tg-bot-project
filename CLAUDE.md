# Telegram-бот — агрегатор/фильтр вакансий (учебный проект)

> Контекст для будущих сессий Claude Code. Кратко. Полная постановка — у владельца.

## Что это
Учебный проект, **не продукт и не монетизация**. Единственный пользователь — владелец.
Цель — впервые руками пощупать **проектирование/архитектуру, бэкенд, БД, работу с AI API**.
hh.ru — осознанно учебный полигон. Боль: листание полотен неподходящих вакансий.

## Кто владелец
Senior QA, переходит в автоматизацию/SDET. Сильная зона — тесты (`pytest`).

## Режим работы: вайб-кодинг (СОБЛЮДАТЬ СТРОГО)
- **Код пишет Claude.** Владелец отвечает за верхнеуровневые решения (что делаем, какая
  архитектура, следующий шаг) и за разбор готового кода.
- Цель владельца — учиться **верхнеуровневым решениям** и **читать/понимать код**, а не
  набивать его руками.
- **Перед/после кода — короткое объяснение «что и зачем написал»**, чтобы владелец понял
  diff. Теория идёт как сопровождение кода, а не как задание написать самому.
- Прямо, по делу, без воды. Лучше 5–7 проработанных пунктов, чем 30 «на всякий».
- **Один путь решения, без вариантов A/B/C.**
- Ошибку в подходе/решении называть прямо.
- HTTP/REST/API-концепции объяснять НЕ надо — только специфику hh/инструментов.
- Объяснения — на русском. Контекст «зачем» важен.
- Работаем через VS Code (расширение `anthropic.claude-code`): правки видны как diff.

## Архитектура
Трёхмодульный конвейер: **источник → фильтр → доставка**. Связывает только `pipeline.py`;
модули НЕ импортируют друг друга. **Ядро — единая Pydantic-модель вакансии = контракт
между модулями.** Каждый источник приводит свой формат к этой модели.

```
src/
  models/     # Pydantic-модель вакансии (от неё зависят все)
  sources/    # hh (сейчас), telegram (потом)
  filters/    # фильтр-правила (общий для всех источников)
  delivery/   # консоль (сейчас), telegram-бот (потом)
  pipeline.py # источник → фильтр → доставка
config/       # критерии фильтрации как ДАННЫЕ, не в коде
tests/        # pytest
scripts/      # разовые скрипты (разведка Ит.0) — НЕ часть конвейера
.gitignore    # .venv/  __pycache__/  .env
```

## Итерации (порядок 0→1 жёсткий, 2–4 переставляемы)
0. **Разведка hh API** — одноразовый скрипт в `scripts/`, увидеть СЫРОЙ ответ.
1. **Ядро конвейера** на одном источнике: Pydantic-модель + фильтр-конфиг
   (ключевые/стоп-слова + порог зарплаты) + доставка В КОНСОЛЬ.
2. **Telegram-доставка** (`aiogram`, бот через @BotFather, сводка в личку).
3. **Интерактивные фильтры** в боте через FSM — слой поверх фильтр-функции.
4. **AI-слой ранжирования** (1 HTTP-запрос к LLM «подходит + причина»).

Не путать: ранжирование (ИИ, Ит.4) ≠ контроль объёма выдачи (пагинация + параметры hh).

## СТАТУС: итерации 0-5 закрыты, бот работает

Источник hh закрыт (403 анонимно, поддержка соискателей прекращена 15.12.2025) —
заменён на Telegram-каналы через Telethon. Смена источника не сломала остальное:
менялся только `sources/`. Ровно ради этого модули и разделяли.

### Что есть
- `sources/` Telethon (техаккаунт) -> `parsing/` регулярки -> `enrichment/` Haiku
  (structured output + prompt caching) -> `filters/` правила + ИИ-оценка ->
  `storage/` SQLite (дедуп + настройки) -> `delivery/` aiogram-бот.
  Связывает `pipeline/`.
- Критерии — из резюме (`profile/resume_parser.py`), а не из конфига. Фильтр
  принимает `Criteria` параметром и не знает, откуда они.
- Резюме отвечает не на всё (ЗП в нём часто нет), поэтому в карточке
  подтверждения поля правятся руками — `profile/inputs.py` + FSM в `bot_ui`.
  Кнопка «Критерии» открывает ту же карточку с данными из БД: перезаливать
  резюме ради смены формата бессмысленно, формата в резюме нет.
- Кнопка «Все вакансии» — архив: не смотрит ни на `seen_at`, ни на возраст
  поста. Без неё показанная вакансия старше недели была недостижима.
- Два процесса: `scripts/daily_fetch.py` (крон, раз в сутки) и
  `scripts/run_bot.py` (бот, слушает кнопки).

### Режим работы: pull с тихим дайджестом
Крон собирает и шлёт ОДНО тихое сообщение "доступно N новых". Новых нет —
МОЛЧИТ. Чтобы тишина не была двусмысленной ("сломался" vs "нечего показать"),
`/start` и "Показать новые" пишут строку `_last_fetch_line`: когда последний
раз ходили и сколько нашли. Ноль — тоже ответ.

Время показывается В МОСКОВСКОМ поясе, а в БД пишется aware UTC. Сервер живёт
в UTC: наивное время показало бы 04:02 вместо 07:02, то есть подтвердило бы
подозрение, что утренний прогон не сработал.

Кнопки ДВУХ сортов, не путать:
- ПОКАЗА (Показать новые, За сегодня/3 дня/неделю, Все вакансии) — только
  читают БД. Мгновенно, ни Telethon, ни ИИ. Обход сюда добавлять НЕЛЬЗЯ:
  после утреннего дайджеста он вернёт то же самое, а ждать придётся.
- ОБХОДА («Проверить сейчас») — единственная, что идёт в сеть. Нужна между
  прогонами крона: посмотрел утренние, написал HR, а после обеда хочешь
  свежие. Установившийся обход ~20 с (дедуп отсекает виденное до ИИ).

**Обход под замком** (`src/pipeline/lock.py`, flock). Крон и бот — разные
процессы с одним `.session`: два обхода разом дают AUTH_KEY_DUPLICATED и отзыв
авторизации техаккаунта. Достаточно нажать кнопку в 7:00 или дважды подряд.
flock, а не флаг в БД: ОС снимает его сама, когда процесс умирает.

### Кнопки vs FSM (грабли, уже наступали)
Кнопки меню приходят обычным текстом, а хендлеры состояний в `bot_ui`
зарегистрированы РАНЬШЕ кнопочных — значит состояние заберёт нажатие себе.
Новую кнопку меню обязательно клади в `MENU_BUTTONS`: фильтр `NOT_MENU_BUTTON`
на хендлерах состояний пропускает её вниз, к кнопочному хендлеру. Кнопки показа
и `/start` при этом гасят состояние — брошенный онбординг не должен съедать
следующее сообщение. Проверено в `tests/test_bot_ui.py` через реальный роутинг.

### Шкала оценки: 4 значения, не 3
`high` / `medium` / `low` / `no_data`. Показываются все, КРОМЕ `low`
(`passed = score is not LOW`).

`no_data` — пост-заглушка: «должность + ссылка на BambooHR», внутри ни языка,
ни стека. Это НЕ medium («язык подходит, но есть расхождение») и НЕ low
(«проверили и мимо»): тут не проверили ничего. В low класть нельзя — low не
показывается, а за ссылкой может быть отличная вакансия. Бейдж
«🔗 НЕТ ОПИСАНИЯ» отличает её от medium с одного взгляда.

Границу проводит НАЛИЧИЕ ОПИСАНИЯ, а не наличие языка — это выяснилось на
живых данных. Пост с обязанностями и требованиями, но без языка, — НЕ no_data:
так выглядит вакансия на РУЧНОЕ тестирование, где язык не назван не от пустоты
поста, а потому что программировать не надо. Это известный факт, а не
неизвестность, и оценивается обычной шкалой (ручное без автоматизации -> low).
Отсутствие ОДНОЙ зарплаты — тоже medium, не no_data.

Первая формулировка («нет технической сути») эту границу не держала: no_data
начал впитывать всё, что не лёг на шкалу, — ровно то, чем раньше болел medium.
Замерено: 2 ложных из 28.

### Переоценка при смене критериев
`vacancies.criteria_hash` — отпечаток критериев, под которыми вакансию
оценивали. Без него протухшую запись от свежей не отличить, и переоценка
платит ИИ за все подряд. В отпечаток идут ТОЛЬКО жёсткие поля (язык, зарплата,
формат, грейд, стоп-стек): инструменты мягкие, их правка платную переоценку
тянуть не должна.

`scripts/reassess.py` — разовый скрипт, по умолчанию показывает диф, пишет с
`--apply`. Прогон 2026-07-15: из 17 записей убрано 9 (чужой язык, зарплата ниже
порога, гибрид при remote-only).

`--all` — переоценить ВСЁ, а не только протухшее. Нужен после правки промпта:
отпечаток следит за критериями, а формулировки шкалы в него не входят. Версию
промпта в отпечаток не вшивали намеренно — в разработке он правится часто, и
каждая мелкая правка тянула бы платную переоценку всего архива.

### БД самодостаточна: `raw_text` лежит в ней
Исходный текст поста хранится в `vacancies.raw_text`. Это не про место (~11 МБ
в год), это про отладку: без него любой разбор лезет в Telegram за исходником,
а значит требует `.session`, который живёт в одном месте. С ним БД — законченный
артефакт: скачал файл с VPS и воспроизвёл что угодно офлайн, `reassess`
сети не требует.

Записи до появления колонки лежат с `raw_text=NULL`: за ними скрипт ещё сходит
в Telegram и тут же допишет исходник — вылечиваются сами.

### Валюта: ИИ извлекает, КОД конвертирует
10% постов-вакансий дают вилку в долларах (замер 2026-07-15: 40 из 390) — это
международная удалёнка, ровно то, что нужно при remote-only.

Было противоречие: промпт обогащения велел «нормализуй в рубли», а схема тут же
предлагала поле `currency` с USD/EUR. Выполнить обе инструкции нельзя, и модель
каждый раз решала по-своему: один и тот же «от 2 500 $» стал то 2 500 000 руб
(применила правило «голое число = тысячи» к долларам), то честными 2500 USD.
А `_salary_reasons` валюту не смотрел и сравнивал `2500 < 230000` — вакансия на
~200к вылетала МОЛЧА.

Теперь: ИИ извлекает как написано (`2500`, `USD`), код конвертирует.
- `config/currency.py` — курсы как ДАННЫЕ. Приблизительные и протухают: нужны
  для сравнения с порогом, а не для бухгалтерии. Незнакомая валюта -> None,
  то есть «неизвестно», а неизвестное правилами не режем.
- `Salary.min_rub`/`max_rub` — СВОЙСТВА модели, не хелпер в вызывающем коде.
  Зарплату читают правило, промпт оценки и карточка; забыть конвертацию в одном
  месте = снова молча терять вакансии. Свойство забыть нельзя.
- `salary_currency` в БД: без неё постфактум не понять, 2500 — рубли или доллары.

### Проектные ограничения (осознанные, не баги)
- Один пользователь, один профиль.
- Опрос каналов раз в сутки — риск бана Telethon-аккаунта; с ростом числа
  подписок растёт и он.
- Бот не пушит вакансии по одной, только дайджест.
- Batch API не используется: латентность до 24ч, а дедуп сводит поток к
  ~80 постам/сутки, где экономия — центы.

### Откуда берутся каналы
Из ПОДПИСОК техаккаунта: подписался в приложении — источник появился, `.env`
править не надо. Приоритет: аргумент `iter_posts` -> `CHANNELS`/`CHANNEL` в
`.env` -> подписки. В `.env` переменная закомментирована — это аварийный выход
на случай, если подписки понадобится проигнорировать.

Две грабли, обе уже наступали:
- **Мегагруппа ≠ broadcast.** `@qa_jobs` — мегагруппа (`broadcast=False`), и
  фильтр «только broadcast-каналы» отсёк бы её молча. Берём тип `Channel`
  целиком: и каналы, и мегагруппы.
- **Имя канала = ключ дедупа** в `seen_posts`. В БД лежит `@qa_jobs`, Telethon
  отдаёт `qa_jobs` без собаки. Всё гоним через `normalize_channel`, иначе ключ
  разъедется и история канала уедет в ИИ по второму разу.

Подписки техаккаунта = список источников, поэтому посторонним подпискам на нём
не место — иначе начнём скрапить их.

### Замеры на 9 каналах (2026-07-15, LIMIT=50, критерии Python/230к/remote)
Прочитано 450 постов -> дедуп снял 129, предфильтр 137 (оба БЕЗ ИИ) ->
обогащено 184 -> правила отсекли 122 -> оценено ИИ 62.
Итог: 5 high, 15 medium, 25 no_data, 17 low (low не показывается).
Вызовов ИИ 246, сэкономлено 532. Время: Telethon 3 с, обогащение 86 с,
фильтр+оценка 34 с.

Главное, что видно из этих цифр: **больше половины потока — посты-указатели**

каналов-агрегаторов. Проверено вручную: из 28 no_data настоящих заглушек 26.

Старый замер на ОДНОМ канале (300 постов, 76 дней): поток ~118 постов/мес, на
пороге 270к проходило ~8 вакансий/мес. Экономия на кэше 88%, ~$0.30 за 300

- Резюме читается из PDF/TXT. Скан (PDF без текстового слоя) не читается —

### Целевой VPS (проверено 2026-07-16)
Ubuntu 24.04.2, Python 3.12.3, systemd 255, пояс **UTC**. Рядом живут XRay (VPN
владельца в Telegram) и TorrServer — не трогать, ляжет XRay = ляжет доступ.
Сам VPS является выходом в Telegram, боту VPN не нужен.

Расписание — systemd-таймер, не крон: пояс задаётся явно
(`OnCalendar=Mon-Fri 07:00 Europe/Moscow`, это 04:00 UTC), а `Persistent=true`
догоняет пропуск, если сервер лежал в 7 утра. Крон бы промолчал.

**После копирования `.session` на VPS локально нельзя запускать обход**
(`daily_fetch.py`, кнопка «Проверить сейчас», `reassess` на старых записях):
один session-файл с двух машин = AUTH_KEY_DUPLICATED и отзыв авторизации.
Отладке сеть не нужна — исходники постов лежат в БД.

### Запуск
    .venv/bin/python scripts/run_bot.py       # бот, слушает кнопки (долгоживущий)
    .venv/bin/python scripts/daily_fetch.py   # сбор, крон 0 20 * * *
    .venv/bin/python scripts/reassess.py      # переоценка архива (диф; --apply пишет)
Отчёт сбора пишется в `logs/daily_fetch.log` (и в stdout). Пишет сам скрипт, а
не редирект в crontab: под кроном stdout уходит в никуда, и молчащий крон
неотличим от «новых нет». Падения тоже идут в лог.
`.env`: API_ID, API_HASH, ANTHROPIC_API_KEY, BOT_TOKEN, CHAT_ID.
CHANNELS не нужен — каналы берутся из подписок техаккаунта.
247 тестов, сеть не трогают: `.venv/bin/python -m pytest tests/ -q`
